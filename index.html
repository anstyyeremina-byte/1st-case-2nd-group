Service Analysis

NBU mandatory requirements and global best practices for BNPL and Bulk Payments

Bulk Payments — Mass (Batch) Payments
Bulk Payments for PrivatBank
The ability to initiate hundreds or thousands of payments in a single API request. A company uploads a file or sends an array of data through the API, and the bank processes them as a single initiation operation requiring only one SCA confirmation. Significantly cheaper and faster than outdated banking file protocols.
🏛️
Mandatory NBU Requirements
Bulk Payments
1
Payment initiation via API — initiation through Open Banking (PISP), support for batch submission
2
Strong Customer Authentication (SCA) — approval of each batch via MFA / OTP / app confirmation
3
Consent management — explicit user approval before execution, ability to revoke access
4
Security & compliance — encryption, fraud monitoring, audit trail (who created / approved the batch)
5
Non-discrimination — TPP access must not be worse than the bank's own channels
⚡
Business Benefits
For PrivatBank clients
✓
Automated payroll and supplier settlements
✓
1 approval instead of N individual ones — significantly less friction for businesses
✓
Clear batch summary — one overview instead of dozens of transactions
✓
Support for automatic retries on failed payments
✓
Faster and cheaper processing than legacy bank file protocols
★
Status Model — Batch and Transaction Statuses
UK Open Banking + Netherlands PSD2
▼
The system must provide clear statuses for both the entire batch and each individual transaction — giving full visibility to the business user and TPP.
Pending

Processing

Completed

Business value

Global Best Practice Features
1
Batch as a Single Entity
UK Open Banking · Barclays · HSBC
▼
A batch is a single logical payment object: multiple instructions processed, authorised, and tracked as one unit.
One API request with a list of beneficiaries (IBAN / accounts), amounts, and payment references
One SCA authorises the entire batch — no need to confirm each payment individually
Once approved, transactions cannot be modified — any change requires a new batch
Limits & validation: max number of transactions, max total amount, per-transaction limits
Business value
Less friction (1 vs N)Faster payrollClear batch summary
2
Multi-authorisation
UK & EU corporate banking · Germany · Netherlands
▼
A batch payment requires approval from multiple users before execution, based on predefined company rules.
Role-based access: creator → approver → final approver
Sequential (step-by-step) or parallel (any 2 of 3 approvers)
Configurable rules: number of approvals required, thresholds (e.g. >10,000 UAH requires 2 approvals)
Batch moves to Processing only after all required approvals are completed
Business value
Reduces fraud riskCorporate governanceInternal financial controls
3
Scheduling
EU corporate banking standards
▼
Scheduling allows users to define when a batch should be executed — immediately or at a specified future date and time.
Immediate execution — processed right after authorisation
Future-dated: user specifies an execution date or time, which is locked after SCA
Business value
Automated recurring paymentsAligned with business cyclesLess manual work
4
Partial Execution & Retries
UK implementations
▼
Allows a batch to be processed even if some transactions fail; retries re-process only the failed ones without resubmitting the entire batch.
Successful payments complete; failed ones do not block the entire batch
Manual retry: select failed transactions → system auto-creates a new batch
Automatic retry (optional): after X minutes following a balance top-up
Business value
No full reprocessingFewer operational errorsEfficient payroll
5
Notifications (Webhooks)
Modern fintech APIs · UK · Global
▼
Webhooks push real-time status updates to TPP systems instead of requiring constant polling.
Event-driven: triggered on status change, processing completion, or failure detection
TPP receives updates immediately — no need to repeatedly query for status
Business value
Real-time UXNo polling overheadFaster failure response
6
Templates & Reusability
EU · UK standards
▼
Templates allow users to save predefined batch structures and reuse them without re-entering all data.
Convert a batch into a template: list of beneficiaries, payment references, default amounts
Select a template → system pre-fills data → fewer errors and manual input
Business value
Faster payrollFewer manual errorsConsistent payment structure

BNPL — Buy Now, Pay Later via Open Banking
BNPL Model for PrivatBank
When a customer selects Open Banking as their payment method, the system (TPP) analyses their transaction history in real time via AISP endpoints, instantly runs a credit scoring, and offers to split the payment into instalments. Far more accurate than traditional forms — the bank sees real income and spending over the last 6–12 months.
🏛️
Mandatory NBU Requirements
BNPL
1
User consent — explicit approval of access to account data and / or payment initiation
2
Strong Customer Authentication (SCA) — covers data access, BNPL agreement approval, and payment initiation
3
Authorized provider — TPP must be authorised by the NBU as AISP and / or PISP
4
Account information via AISP — balance, transaction history, income patterns, spending behaviour
5
Transparency — total amount, instalment schedule, fees / penalties, consequences of late payment
6
Data protection & Audit trail — records of consent, SCA, agreement, initiation, and repayment status changes
7
Risk & affordability checks — credit limit validation, behaviour analysis, scoring, internal credit policy
8
Revocation handling — user must be able to revoke Open Banking consent
💳
Business Benefits
For PrivatBank clients
✓
Instant decision with no paperwork — zero friction at checkout
✓
More accurate scoring — real income and spending data instead of a form
✓
Access for thin-file customers with no credit history
✓
Flexible plan choice: Pay in 3 / Pay in 4 / monthly instalments
✓
Automated repayments (mandate) — fewer missed payments
✓
Seamless checkout — higher conversion rate for merchants
★

Overdue

Defaulted

Global Best Practice Features — BNPL
1
Instant Credit Decision
UK & EU Open Banking lending models
▼
Creditworthiness evaluation and BNPL approval in real time (seconds) without any manual review.
Decision made at checkout within seconds — no offline processing or waiting
Data sources: account balance, transaction history, income patterns (salary detection), spending, existing obligations
Decision engine calculates: affordability, risk score, and available credit limit
Business value
Instant decisionNo paperworkSeamless checkout
2
Single Consent + SCA
UK Open Banking / PSD2
▼
A single SCA confirmation covers BNPL agreement acceptance, repayment mandate setup, and first payment authorisation.
One authorisation: BNPL agreement + repayment mandate + first payment
Combined consent: access to financial data + agreement to loan terms + initiation of future payments
Business value
Fewer stepsTransparent agreementFaster decision-to-payment
3
Data-driven Credit Scoring
EU · PSD2 + Open Banking
▼
Uses real financial data from bank accounts to assess creditworthiness instead of relying solely on traditional credit bureaus.
Analysis: income (salary detection), spending patterns, balance stability
Generates a real-time risk score and credit limit
Business value
Higher approval accuracyThin-file usersLower default rates
4
Flexible Repayment Plans
UK / Global · Klarna
▼
Users can choose how to repay: preset plans or monthly instalments.
Preset plans: Pay in 3 / Pay in 4 / monthly instalments
System automatically calculates instalment size and repayment schedule
Business value
Better affordabilityHigher conversion rate
5
Automated Repayments (Mandates)
EU · SEPA / Open Banking
▼
Instalment payments are collected automatically after a single user consent.
User approves the mandate once
System initiates payments automatically according to the repayment schedule
Business value
Fewer missed paymentsBetter repayment rateLower operational costs
6
Seamless Checkout Integration
UK BNPL UX standards
▼
BNPL is offered directly in the checkout flow — alongside other payment methods.
Minimal steps: select BNPL → confirm → done
API / SDK integration into e-commerce platforms and POS systems
Business value
Less frictionHigher checkout conversionBroader distribution
7
Notifications (Real-time Updates)
Klarna · Afterpay · Global fintech APIs
▼
Real-time or scheduled alerts about payments, obligations, and status changes.
Event-driven: payment events, time-based (due dates), status changes
Multi-channel: push (mobile app), SMS (critical alerts), email (summaries / confirmations)
Business value
Fewer late paymentsFinancial awarenessUser trust
8
Limits & Risk Control
UK & Australia regulations
▼
Dynamic credit limits, real-time risk evaluation, and continuous usage monitoring.
Dynamic limits: calculated from financial behaviour, updated over time
Risk-based approvals: each request evaluated by risk score, transaction size, and user behaviour
Usage monitoring: outstanding balance, number of active agreements, repayment behaviour
Business value
Prevents over-indebtednessResponsible lending
